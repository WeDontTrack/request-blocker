# webblock

A Linux website/IP blocker. Blocks outbound traffic to specified domains and IP addresses at the kernel packet layer, so it also stops tools like `curl` (including `curl http://<ip>` and `curl --resolve`), not just browsers.

- **Root-added rules apply to all users** (system-wide).
- **User-added rules apply only to that user**, and users cannot see each other's configuration.
- **No setuid binary.** A small privileged daemon does the work; the CLI is fully unprivileged and is authenticated by the kernel.

> Linux only. The daemon relies on `nftables`/`iptables` and Linux peer credentials (`SO_PEERCRED`). It will not function on macOS/Windows.

## How it works

```
webblock (unprivileged CLI)  --JSON over /run/webblock.sock-->  webblockd (root daemon)
                                                                 |
                  identity via SO_PEERCRED (kernel-verified) <---+
                                                                 |
                              /var/lib/webblock  <--- config store
                                                                 |
                              nftables / iptables+ipset  <--- egress drop rules
```

- The **CLI** (`webblock`) holds no privileges. It packages your request as JSON and sends it to the daemon's Unix socket.
- The **daemon** (`webblockd`) runs as root. For every connection it reads the caller's real uid/gid from the kernel via `SO_PEERCRED` (which cannot be forged) and enforces access control centrally:
  - uid 0 (root) manages the global list and may target any user with `--user`.
  - any other uid may manage only its own list and can never see others'.
- Domains are resolved to IP addresses and inserted into firewall sets; matching is done on the destination IP, which is why raw-IP `curl` requests are blocked too. Because CDN addresses drift, the daemon re-resolves on a configurable interval (default 15m) and re-applies atomically.

### Backend selection

At startup the daemon probes for a working backend, preferring **nftables** (table `inet webblock`, named hash sets, per-uid `meta skuid` rules) and falling back to **iptables + ipset** (`hash:ip` sets, a `WEBBLOCK` chain hooked from `OUTPUT`, per-uid `-m owner --uid-owner` rules). Hash sets keep lookups ~O(1) even with large blocklists.

## Build

Requires Go 1.24+. From this directory:

```bash
make build            # builds bin/webblock and bin/webblockd for linux/amd64
make GOARCH=arm64 build
```

## Install (on the target Linux host, as root)

```bash
sudo make install
sudo systemctl daemon-reload
sudo systemctl enable --now webblockd
```

This installs `webblock` to `/usr/local/bin`, `webblockd` to `/usr/local/sbin`, and the systemd unit. Blocklists persist in `/var/lib/webblock`.

## Usage

```bash
# Add (domains and/or IPs; comma-separated lists supported)
webblock --add --name example.com,ads.example.net
webblock --add --ip 93.184.216.34
webblock --add --name tracker.com --ip 10.0.0.5,10.0.0.6

# Remove
webblock --remove --name example.com
webblock --remove --ip 10.0.0.5,10.0.0.6

# Show what is blocked (and visible) to you
webblock --show
webblock --show --json        # machine-readable

# Daemon health / active backend
webblock --status

# Version
webblock --version
```

### Scopes

- **As a normal user:** all of the above operate on *your own* blocklist. `--show` displays your list plus the administrator's global list (read-only).
- **As root:** add/remove operate on the **global** list by default. Target a specific user with `--user <name|uid>`:

  ```bash
  sudo webblock --add --name facebook.com            # global (everyone)
  sudo webblock --add --user alice --name x.com      # only alice
  sudo webblock --show                               # global + every user's list
  ```

## Uninstall / remove all rules

```bash
sudo make uninstall          # stops service, tears down firewall, removes files
# or just remove the firewall rules while keeping the service:
sudo webblockd -teardown
```

## Security notes

- The only privileged component is the daemon, and its interface is a small, validated JSON protocol over a Unix socket. There is no setuid binary.
- Caller identity comes from `SO_PEERCRED`, read by the kernel, so a client cannot impersonate another user.
- Inputs are normalized and validated (domain syntax, IP parsing, list-size limits) and requests are rate-limited per uid. Mutations are audit-logged to the journal with the requesting uid and target scope.

## Known limitations

- IP-based enforcement of a domain can briefly lag a CDN address rotation until the next re-resolution; tune with `--refresh`.
- Per-user matching uses the socket-owner uid; traffic generated by root-owned helper processes on a user's behalf is attributed to that helper's uid.
- The daemon must run as root (or with `CAP_NET_ADMIN`) to program the firewall.

## Project layout

```
cmd/webblock        unprivileged CLI client
cmd/webblockd       privileged daemon, split by concern:
  main.go             wiring + flags
  config.go           Config struct + Logger (all tunables in one place)
  identity.go         peerCred (kernel SO_PEERCRED identity)
  server.go           transport: socket listen/accept/decode/write
  ratelimit.go        per-uid fixed-window limiter
  authz.go            scope resolution + access control
  dispatch.go         action -> Handler routing + protocol version check
  handlers.go         add/remove/show/status handlers
  engine.go           store + resolver cache + backend orchestration
internal/proto      JSON IPC types (versioned)
internal/store      config persistence (/var/lib/webblock)
internal/validate   normalization + validation
internal/backend    Backend interface + nftables and iptables implementations
internal/resolver   domain -> IP resolution (concurrent)
deploy/             systemd unit
docs/               DESIGN.md, STANDARDS.md
```
